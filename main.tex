\begin{frame}{One-sided cliffs fool the trough kernel}
\begin{columns}
\column{0.4\linewidth}
\includegraphics[width=\linewidth]{img/kernel-cliff-veto.png}
\column{0.6\linewidth}
\begin{itemize}
\item Convolutions with the trough kernel also fire on one-sided cliffs --- usually artificial cliffs along roadsides --- since a cliff matches one side of a trough.
\item As a veto, we add another kernel that only matches one-sided cliffs: a positive Gaussian on one side of an angled line, negative on the other.
\item Positive weight on trough convolutions and negative weight on cliff convolutions distinguishes troughs from cliffs.
\end{itemize}
\end{columns}
\end{frame}

\begin{frame}{Developing a model}
\begin{itemize}
\item This reasoning ultimately leads to a Convolutional Neural Network (CNN).
\begin{itemize}
\item Rather than building a set of convolution kernels by hand, they'd be learned.
\item As a supervised learning technique, a CNN would require hand-labeled training data.
\end{itemize}
\item We can label some gullies by hand, but not enough to train a CNN.
\includegraphics[width=0.3\linewidth]{img/manual-labeling.png}
\item Compromise:
\begin{itemize}
\item We hand-draw gully labels for training data (in-gully, out-of-gully, and unknown) in 9 watersheds.
\item We engineer convolution kernels as features (different segment lengths; anti-cliff veto).
\item Perform logistic regression, which is equivalent to a single CNN layer (with sigmoid activation).
\end{itemize}
\end{itemize}
\end{frame}

\begin{frame}{Convolving hundreds of gigabytes with unusually large kernels}
\begin{itemize}
\item Typical image convolution kernels are $3\times 3$ through $11\times 11$ pixels.
\item Ours need to be at least 50 meters on a side to average over enough surrounding elevation: $101\times 101$ pixels.
\item \texttt{scipy.signal.convolve2d} was too slow for exploratory work at county scale.
\item GPU drop-in replacements from CuPy and JAX were faster, but still not the fastest.
\end{itemize}

\begin{frame}{A custom CUDA kernel with Numba}
\begin{itemize}
\item The fastest option: a custom CUDA kernel compiled by Numba.
\item Fix the kernel size at compile time and simplify boundary conditions.
\item Process one watershed at a time: pad, convolve, then clip to the watershed (gullies cannot cross a watershed divide).
\end{itemize}

\begin{frame}{Numba for skeletonization, too}
\begin{itemize}
\item Scikit-Image's skeletonization methods apply to bitmaps, discarding the gully-probability information.
\item We implemented a distance-ordered homotopic thinning algorithm (inspired by Pudney) as a custom Numba CUDA kernel.
\item Pixels are eroded greedily in order of increasing probability, rather than distance.
\item Connectivity is preserved as a constraint, ensuring that all gullies remain trees.
\end{itemize}
\end{frame}

\begin{frame}{Technical highlight}
\begin{columns}
\column{0.55\linewidth}
\begin{itemize}
\item For performance, the visible layers are delivered on demand as tiles.
\item No server required: the PMTiles file format is addressable one tile at a time, served statically from Cloudflare.
\item The ``volume of erosion'' and ``elevation along line'' features also use PMTiles to download elevation data one tile at a time.
\begin{itemize}
\item But elevation data needs float32 precision, and PMTiles must be RGBA images.
\item So we bit-cast the four 8-bit channels as a 32-bit float.
\end{itemize}
\end{itemize}
